# Heartbeat run-state tracking mis-marks every run as failed

## Symptom

`memory/cron-state.json` shows the `heartbeat` entry with:
- `last_status: "failed"`
- `total_runs: 33`, `total_successes: 0`, `total_failures: 33`
- `consecutive_failures: 33`, `success_rate: 0`
- **no `last_success` field at all**

This trips four P0 heartbeat checks simultaneously (failed skill, API degradation
`consecutive_failures >= 3`, chronic failure `success_rate < 0.5`, and the self-check
for a missing `last_success`).

## Why it's almost certainly a tracking artifact, not a real failure

Heartbeat demonstrably runs to completion — it regenerated `docs/status.md` and is
executing this very check. A skill that genuinely failed 33 times in a row would not
be producing output every cycle.

The `last_error` value is a truncated JSON fragment, not an error message:

```
0b53fb9","total_cost_usd":0,"usage":{"input_tokens":0,"cache_creation_input_tokens":0,
"cache_read_input_tokens":0,"output_tokens":0,"server_tool_use":{...
```

This is the tail of a Claude Code result object (prefixed with what looks like a git
short-hash, `0b53fb9`). The `total_cost_usd: 0` and all-zero token usage indicate the
state-update step captured the **wrong line / empty result** when deciding pass-vs-fail
and when extracting the error signature.

`bankr-token-research` carries the same style of garbled `last_error` signature
(`138bdde","total_cost_usd":0,...`) even though its `last_status` is `success` — further
evidence the error-extraction logic is generic and broken, independent of true outcome.

## Impact

Fleet health monitoring is unreliable: the success-detection / state-update step can
mark a successful run as failed and writes a non-informative error signature. Because
heartbeat keys its DEGRADED verdict off these fields, the status page will read DEGRADED
indefinitely until this is fixed, and real failures could be masked by the noise.

## Suggested investigation

- Inspect the workflow step that writes `cron-state.json` (success/failure detection +
  `last_error` extraction). It likely greps the last line of Claude Code stream-json
  output and treats a zero-usage/zero-cost result as failure, or parses the wrong field.
- Confirm whether the heartbeat job's exit code is actually non-zero (e.g. a failing
  auto-commit/push step at the end) vs. the detection logic misreading a successful run.
- Once fixed, reset `heartbeat` counters in `cron-state.json` so historical noise
  doesn't keep tripping P0.
